⚡ ZenvX OS

A voice-driven, Arch-based Linux session with an always-on LLM agent, safety-gated system access, and universal app compatibility.

31 tests passing Arch-based Rust Online-first

Quick Start

Install dependencies (Arch Linux)

sudo pacman -S --needed base-devel rust qemu-system-x86 qemu-ui-gtk grub xorriso dosfstools archiso

Clone & build

git clone https://github.com/sk-deb/ZenvX-OS-prototype01.git
cd ZenvX-OS-prototype01
cargo build --release
cargo test

Run the TUI (no ISO needed)

cargo run -p zenvx        # first-boot: choose provider
cargo run -p zenvx-tui    # launch the interface

See it boot in QEMU

make run       # opens a QEMU window with the ZenvX boot screen
make verify    # headless verification

Build the Bootable ISO

cd iso
./build-iso.sh    # outputs iso/iso-out/*.iso (~1.9GB)

Flash to USB

⚠️ This erases the target device. Double-check the device name with lsblk.
# Identify your USB
lsblk -o NAME,SIZE,TYPE,TRAN,RM

# Flash (replace /dev/sdX)
sudo umount /dev/sdX?* 2>/dev/null
sudo dd if=iso/iso-out/*.iso of=/dev/sdX bs=4M conv=fsync status=progress
sync

Boot from USB

  1. Plug the USB into the target machine
  2. Enter the boot menu (F12 / F2 / Esc / Del)
  3. Select the USB device
  4. First boot asks for your OpenRouter API key (or switch to local Ollama)
  5. Drops into the ZenvX TUI — full-screen chat + app launcher

Architecture

CrateRole
commonShared types, errors, secure config (0600 perms)
ai-coreStreaming LlmProvider; OpenRouter (SSE) + Ollama (ndjson)
agentTool-calling + safety gate (double-confirm, hard denylist)
replTerminal LLM REPL with inline tool execution
launcherApp compat router (.exe→Wine, AppImage, Flatpak, pacman, VM)
voiceWake→STT→TTS pipeline; degrades if whisper/piper absent
modeHardware detection → online-first / local / VM policy
compositor80/20 zone layout + surface tiling engine
shell-uiSettings state + API key masking
sessionWires voice → agent → launcher → UI → compositor
tuiFull-screen terminal interface (ratatui)

Safety

  • Catastrophic commands (rm -rf /, mkfs, fork bombs) → hard-denied, never executed
  • Root/destructive commands (sudo, rm -r, shutdown) → double confirmation required
  • App names → single-quoted to prevent shell injection
  • API keys → stored with 0600 perms, masked in UI, never in the repo
